# vue-fastapi-admin 项目技术文档

## 1. 项目概述

`vue-fastapi-admin` 是一个基于 FastAPI + Vue3 的前后端分离后台管理系统模板，核心目标是提供可快速落地的管理后台基础能力：

- 用户认证与授权（JWT）
- RBAC 权限模型（用户-角色-菜单/API）
- 动态菜单与动态路由
- 系统管理模块（用户、角色、菜单、API、部门）
- 审计日志

项目可用于中小型业务系统的管理端快速搭建，也适合作为 FastAPI + Vue3 的工程化参考。

## 3. 目录结构与职责

## 3.1 根目录

- `run.py`：后端启动入口（uvicorn）。
- `app/`：后端主代码。
- `web/`：前端主代码。
- `deploy/`：部署配置与示意资源。
- `Dockerfile`：容器构建。

### 3.2 后端目录

- `app/__init__.py`：应用工厂，注册中间件、异常处理、路由、生命周期。
- `app/api/v1/`：按模块拆分的接口层。
- `app/controllers/`：业务控制器层，封装 CRUD 与业务逻辑。
- `app/models/`：Tortoise ORM 模型层。
- `app/schemas/`：Pydantic 请求/响应结构。
- `app/core/`：中间件、依赖注入、异常处理、通用 CRUD 等核心设施。
- `app/settings/`：配置。
- `app/utils/`：JWT 与密码工具。

### 3.3 前端目录

- `web/src/main.js`：前端入口。
- `web/src/router/`：静态路由、动态路由注入、守卫。
- `web/src/store/`：Pinia 状态管理（用户、权限、标签页、应用状态）。
- `web/src/api/index.js`：前端 API 映射。
- `web/src/utils/http/`：Axios 封装与拦截器。
- `web/src/views/`：页面（登录、系统管理、错误页、工作台等）。
- `web/src/components/`：复用组件（CRUD 表格、弹窗、页面容器等）。

## 4. 后端架构设计

### 4.1 应用启动流程

1. 运行 `run.py` 启动 uvicorn，加载 `app:app`。
2. `app/__init__.py` 创建 FastAPI 应用。
3. 注册中间件（CORS、后台任务、审计日志）。
4. 注册异常处理器与路由。
5. 进入 lifespan：执行初始化数据逻辑。

### 4.2 初始化逻辑（`app/core/init_app.py`）

`init_data()` 顺序执行：

1. `init_db()`：初始化数据库与迁移。
2. `init_superuser()`：首次启动创建管理员。
3. `init_menus()`：初始化系统菜单。
4. `init_apis()`：扫描并同步 API 资源。
5. `init_roles()`：初始化管理员/普通用户角色及权限。

### 4.3 路由组织

- 根路由：`/api`
- 版本路由：`/api/v1`

模块：

- `base`：登录、用户信息、用户菜单、用户 API、修改密码。
- `user`：用户管理。
- `role`：角色管理与授权。
- `menu`：菜单管理。
- `api`：API 资源管理与刷新。
- `dept`：部门管理。
- `auditlog`：审计日志查询。

除 `base` 外，默认通过 `DependPermission` 进行鉴权与权限校验。

### 4.4 分层说明

- API 层：参数解析、调用控制器、返回统一响应体。
- Controller 层：领域逻辑、模型聚合操作。
- Model 层：数据结构与关系定义。
- Schema 层：输入输出校验。

### 4.5 统一响应结构

通过 `app/schemas/base.py` 返回：

- `Success`：`{code, msg, data}`
- `Fail`：`{code, msg, data}`
- `SuccessExtra`：分页场景附加 `total/page/page_size`

前端以 `code === 200` 判定成功。

## 5. 鉴权与权限体系

### 5.1 JWT 鉴权

- 登录接口：`POST /api/v1/base/access_token`
- Token 通过请求头 `token` 传递。
- JWT payload 包含：`user_id`、`username`、`is_superuser`、`exp`。

### 5.2 认证依赖

`AuthControl.is_authed`：

- 解析 token 并查找用户。
- 将当前用户 ID 写入 `CTX_USER_ID` 上下文。
- 对异常 token 或过期 token 返回 401。

### 5.3 权限依赖

`PermissionControl.has_permission`：

- 超级管理员直接放行。
- 普通用户从角色中汇总授权 API 集合。
- 使用 `(method, path)` 精确匹配当前请求。

### 5.4 前后端权限联动

- 后端提供 `/base/usermenu` 和 `/base/userapi`。
- 前端权限 store 将菜单树转换成路由，并缓存 API 权限集合。
- 页面按钮通过 `v-permission` 指令控制显隐和操作入口。

## 6. 数据模型设计（核心）

定义于 `app/models/admin.py`。

### 6.1 用户（User）

- 基础信息：用户名、邮箱、手机号、密码、激活状态、超级管理员标识等。
- 关系：与角色 `Role` 多对多。
- 组织：`dept_id` 表示部门。

### 6.2 角色（Role）

- 字段：名称、描述。
- 关系：与菜单 `Menu` 多对多、与 API `Api` 多对多。

### 6.3 菜单（Menu）

- 字段：类型、图标、路径、父级、组件、排序等。
- 结构：通过 `parent_id` 构建树形菜单。

### 6.4 API 资源（Api）

- 字段：path、method、summary、tags。
- 用于权限控制与角色授权。

### 6.5 部门（Dept/DeptClosure）

- `Dept`：部门基础信息（含软删除标记）。
- `DeptClosure`：闭包表关系（祖先-后代-层级）。

### 6.6 审计日志（AuditLog）

- 记录用户、模块、接口摘要、请求路径、状态码、耗时、请求参数与返回体。

## 7. 中间件与异常处理

### 7.1 中间件

#### 1) CORS 中间件

- 配置来源：`app/settings/config.py`
- 默认允许跨域（`*`）

#### 2) 后台任务中间件（BackGroundTaskMiddleware）

- 请求前初始化任务容器。
- 请求后统一执行积压任务。

#### 3) 审计日志中间件（HttpAuditLogMiddleware）

- 记录 GET/POST/PUT/DELETE 请求。
- 可配置排除路径。
- 自动提取路由标签与 summary。
- 支持大响应体截断保护（默认 1MB）。

### 7.2 异常处理

`app/core/exceptions.py` 统一处理：

- ORM 不存在异常（404）
- ORM 完整性异常（500）
- HTTP 异常（透传状态码）
- 请求/响应校验异常

## 8. 前端架构设计

### 8.1 初始化流程

1. `main.js` 创建 Vue 应用。
2. 注入 Pinia。
3. 初始化路由并尝试动态路由注入。
4. 注入指令、i18n、全局能力后挂载。

### 8.2 路由体系

- 静态路由：工作台、登录、错误页等。
- 动态路由：登录后从后端菜单构建并 `router.addRoute` 注入。
- 无 token 时注入空路由兜底。

### 8.3 状态管理

`store/modules`：

- `user`：用户信息、登出流程。
- `permission`：菜单路由与 API 权限集合。
- `tags`：多标签导航状态。
- `app`：布局与显示状态。

### 8.4 请求机制

- Axios 在请求拦截器中自动注入 token。
- 响应拦截器统一处理 `code !== 200` 与网络异常。
- `401` 自动清理状态并跳转登录。

## 13. 风险与优化建议

### 13.1 安全类

1. `SECRET_KEY` 当前为代码内明文，建议改为环境变量注入。
2. 默认管理员账号密码（`admin/123456`）仅适合开发演示，生产必须首次强制修改。
3. `token == "dev"` 的认证分支建议仅在开发模式启用，并默认关闭。

### 13.2 数据与模型类

1. `User.dept_id` 目前为普通整数字段，可考虑升级为外键并增加约束。
2. 部门闭包表更新逻辑建议补充回归测试，确保改父级场景的关系正确性。

### 13.3 工程类

1. 增加后端单元测试与接口测试（鉴权、权限、初始化迁移、部门树）。
2. 增加前端关键页面 E2E 测试（登录、动态路由、权限按钮、用户与角色管理）。
3. 对初始化脚本增加环境保护，避免生产环境误执行破坏性操作。
